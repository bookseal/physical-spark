Sim setup · Mission 0

Pat me on the back

Set up a robot arm in simulation, make it gently sweep — a pat on the back — and record a short clip. Your first win.

Beginner ~1 hour Mac (Apple Silicon) Simulation only
What you'll build: A simulated robot arm that sweeps left and right — like a pat on the back — saved as a short video you can keep. No real hardware needed. The point is not skill; it's the warm feeling of a robot you set up yourself.

1 Check your Python ▶

Every step here uses Python. Your Mac already has it. Let's make sure.

Open the Terminal app (press ⌘ + Space, type "Terminal", hit Enter). Then type:

python3 --version

✓ Checkpoint

You should see a line like Python 3.9.6 (any 3.x is fine). If you see a version number, Python works.

Why python3 and not just python?
On older Macs, python meant the old Python 2. Apple removed Python 2, so today only python3 exists by default. Many tutorials still write python — on a Mac, use python3. (In Step 2 we make a fresh environment where plain python works again.)
2 Create your environment ▶

We keep this project in its own clean "box" so it can't break anything else on your Mac. This box is called a conda environment. (If you don't have conda yet, install Miniforge first, then come back.)

Make the box and step inside

conda create -y -n lerobot python=3.12
conda activate lerobot
python --version

✓ Checkpoint

Your prompt now starts with (lerobot), and python --version prints Python 3.12.x. Now plain python works.

If conda activate says "run conda init": run conda init zsh, then open a new Terminal window and try conda activate lerobot again. It only works in a fresh window.
What does conda activate actually do?
It puts your environment's folder at the front of the search path. When you type python, your Mac looks front-to-back and finds this box's Python 3.12 first — before the system one. Nothing is deleted; conda deactivate puts it back.
3 Get LeRobot ▶

Now we download LeRobot (the robot toolkit) and install it, together with two simulators.

git clone https://github.com/huggingface/lerobot.git
cd lerobot
pip install -e ".[pusht,aloha]"

This install takes a few minutes and prints a lot of text — that's normal. Tip: run git clone from a folder like your home directory, not from inside another folder named lerobot, so you don't end up with lerobot/lerobot.

✓ Checkpoint

Test that it installed. This should print nothing and show no error:

python -c "import lerobot"
What do the [pusht,aloha] parts mean?
They are extras — optional add-ons. "Install LeRobot, and also the parts I need for pusht and aloha." aloha is the two-arm robot simulator we'll use. Robot libraries are big, so you only install the pieces you need.
4 Meet the arm ▶

Let's take one picture of the arm before it moves. Create a file named hello_sim.py inside the lerobot folder and paste this in:

"""hello_sim.py — take one picture of the arm's starting pose."""
import gym_aloha                 # importing this registers the arm world
import gymnasium as gym
import imageio.v2 as imageio

# "rgb_array" = give me pixels, do not open a window
env = gym.make("gym_aloha/AlohaTransferCube-v0", render_mode="rgb_array")
env.reset(seed=0)               # same start every time
frame = env.render()            # one picture: 480 x 640 x 3 (RGB)
imageio.imwrite("arm_hello.png", frame)
env.close()
print("saved arm_hello.png")

Run it, then open the picture:

python hello_sim.py
open arm_hello.png

✓ Checkpoint

The terminal prints saved arm_hello.png. The picture shows two white grippers facing each other and a red cube in the middle.

What is that env thing?
An environment — one whole little world: the arms, the cube, and the physics, all in one object. env.reset() starts it, env.render() draws it. You'll use these same words for almost every robot simulator.
5 Make it sweep (the pat) ▶

Now the fun part: a live window where the arm moves in real time. Create live_sim.py and paste this:

"""live_sim.py — open a live window and watch the arm pat.
On macOS you MUST run this with mjpython (not python)."""
import time
import numpy as np
import gym_aloha
import gymnasium as gym
import mujoco.viewer

# No render_mode here: the window is drawn by the viewer below.
env = gym.make("gym_aloha/AlohaTransferCube-v0")
env.reset(seed=0)

# reach the real physics objects inside the env
phys = env.unwrapped._env.physics
model, data = phys.model.ptr, phys.data.ptr

with mujoco.viewer.launch_passive(model, data) as viewer:
    step = 0
    while viewer.is_running():
        wave = np.sin(step * 0.05)                    # slow back-and-forth, -1..1
        action = np.full(14, wave, dtype=np.float32)  # the pat: sweep both arms
        env.step(action)                              # move one small step
        viewer.sync()                                 # show the new pose
        time.sleep(0.02)                              # ~50 fps
        step += 1
env.close()

Run it with mjpython — this is the special launcher, not python:

mjpython live_sim.py

The loop that makes it move:

pick action (sweep)→ env.step()→ viewer.sync()↺ repeat

✓ Checkpoint

A window opens and the arms sweep gently left and right — the pat. Drag with the mouse to rotate the view. Close the window to stop.

If it crashes or won't open: (1) You must use mjpython, not python. (2) Make sure (lerobot) is active. (3) Run it in the real Terminal app. A NSWindow ... main thread crash usually means render_mode was set — this file leaves it out on purpose.
Why mjpython instead of python?
macOS only lets the main thread open windows. mjpython arranges the window on the main thread for you. Plain python can't, so the window fails to open. (For saving a video with no window, in the next step, plain python is fine.)
Want to see random flailing instead?
Replace the two wave/action lines with action = env.action_space.sample(). Now the arms twitch randomly — because each step is fresh random numbers, with no plan. The sine wave is your first tiny "plan".
6 Record your clip ▶

Last step: save the pat as a video. Create sim_arm_demo.py:

"""sim_arm_demo.py — record the pat as a video (no window)."""
import numpy as np
import imageio.v2 as imageio
import gymnasium as gym
import gym_aloha

ENV_ID = "gym_aloha/AlohaTransferCube-v0"
N_STEPS = 200
OUT_PATH = "sim_arm.mp4"

def choose_action(step):
    """The pat motion: sweep both arms with a slow sine wave."""
    wave = np.sin(step * 0.05)
    return np.full(14, wave, dtype=np.float32)

env = gym.make(ENV_ID, render_mode="rgb_array")
env.reset(seed=42)
frames = [env.render()]                 # first frame
for step in range(N_STEPS):
    env.step(choose_action(step))
    frames.append(env.render())         # save each frame
env.close()
imageio.mimsave(OUT_PATH, frames, fps=30)
print(f"saved {len(frames)} frames -> {OUT_PATH}")

Run it (plain python is fine — no window this time), then watch:

python sim_arm_demo.py
open sim_arm.mp4

✓ Checkpoint

The terminal prints saved 201 frames -> sim_arm.mp4, and the clip plays: your arm giving a slow pat. That's your win. 🎉

Why two ways to see it — window vs video?
Live window (Step 5) is for watching and debugging. Video / no window (this step, called "headless") is how training works on servers with no screen — fast, and easy to save or share. Same arm, two purposes.

Quick recap questions

env.reset() gives back a picture (obs) but no reward. Why?
Because reward is the result of an action, and reset hasn't taken any action yet — the round just started. env.step() is the one that gives a reward, after you act.
Why is every arm command a number between -1 and 1?
Real joints have different ranges (angles, gripper width…). Putting them all on the same -1..1 ruler keeps things simple and stable. The simulator turns -1..1 back into real angles inside.
What would make this a "real" robot brain, not a sine wave?
A policy: a function that looks at the picture (obs) and chooses the action — obs → action. Our sine wave ignores what it sees. A trained policy (a neural network) would watch the cube and act on it. That's the next mission.
