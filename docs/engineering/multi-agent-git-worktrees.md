# 여러 AI 에이전트를 브랜치·폴더로 나눠 병렬로 굴리기 (git worktree)

> 대상: 창업 프로세스·git 초심자. 한 프로젝트에서 Claude Code 세션 여러 개를
> 동시에 돌리되, 서로 파일을 밟지 않게 **폴더와 브랜치로 격리**하는 법.
> 그리고 창님이 특히 궁금해한 **"각 브랜치는 늘 main을 pull 해야 하나?"** 에 대한 답.

---

## 0. 한 장 요약 (먼저 그림부터)

```
                한 개의 저장소(.git 히스토리는 공유)
                          │
   ┌──────────────────────┼──────────────────────┐
   ▼                      ▼                      ▼
폴더: physical-spark      physical-spark-game    physical-spark-video
브랜치: main              feature/game-fix       feature/hero-video
세션: (배포/통합)         claude 세션 A          claude 세션 B
```

- **세션(에이전트) = 브랜치 = 폴더**, 이 셋을 1:1:1로 맞추는 게 전부입니다.
- 폴더가 물리적으로 갈리니 두 에이전트가 **동시에** 파일을 고쳐도 안 부딪힙니다.
- 히스토리(`.git`)는 하나를 공유하므로, 한 폴더에서 `git fetch` 하면 **모든 폴더가 최신 커밋 정보를 같이** 봅니다.

---

## 1. 용어 3개만 (이것만 알면 됨)

| 용어 | 초심자용 풀이 |
|---|---|
| **branch(브랜치)** | 커밋(작업 저장점)들의 한 갈래. "main에서 갈라져 나온 실험 선(線)". GitHub는 이걸 push하면 원격에 복사본이 생기는 것뿐. |
| **worktree(워크트리)** | 한 저장소를 **여러 폴더에 동시에 펼쳐 놓는** 기능. 폴더마다 다른 브랜치를 checkout해 둘 수 있어, 브랜치를 왔다갔다 갈아끼울 필요 없이 병렬 작업이 됨. |
| **PR(Pull Request)** | "내 브랜치를 main에 합쳐주세요" 요청서. 변경 이력이 남고, 합치기 전에 한 번 검토하는 관문. |

> 보통 `git checkout other-branch`는 **한 폴더 안에서** 브랜치를 바꿔 끼웁니다(동시에 둘 못 함).
> worktree는 **폴더 자체를 하나 더** 만들어 그 문제를 없앱니다.

## 2. 셋업: 폴더+브랜치 나누기

### 방법 A — Claude Code에게 맡기기 (가장 쉬움)
```bash
claude -w hero-video        # 새 worktree 폴더 + 브랜치를 만들고, 그 안에서 세션 시작
```
`-w`(=`--worktree`)는 워크트리 폴더와 브랜치를 만들고 그 폴더에서 세션을 엽니다.
여기에 `-n`으로 이름까지 붙이면 나중에 `claude -r` 목록에서 구분이 쉽습니다:
```bash
claude -w hero-video -n hero-video
```

### 방법 B — 직접 git으로 (원리 이해용)
```bash
# 현재 위치: /Users/chan/workspace/physical-spark  (= main 폴더)
git worktree add ../physical-spark-hero-video -b feature/hero-video
#            └ 새로 만들 폴더 경로 ┘          └ 새 브랜치 이름 ┘

cd ../physical-spark-hero-video    # 이 폴더는 이제 feature/hero-video 브랜치
# 여기서 claude를 열면 이 세션은 이 브랜치에만 영향을 줌
```

### 지금 어떤 워크트리가 있나 보기
```bash
git worktree list
```

## 3. 하루 흐름 (에이전트 2~3개 병렬)

1. main 폴더는 **통합/배포 전용**으로 두고, 거기선 직접 작업하지 않는다.
   (이 repo는 **GitHub Pages가 main을 그대로 라이브 배포** → main은 늘 "깨지지 않은 상태"여야 함.)
2. 작업마다 워크트리를 하나 판다: `hero-video`, `game-fix`, `naming` …
3. 각 폴더에서 각자 Claude 세션이 돈다. 서로 안 밟는다.
4. 한 덩어리가 끝나면 그 브랜치를 push → **PR → main에 merge**.
5. merge된 워크트리는 정리(4단계 아래 "정리" 참고)하고, 다음 작업은 새 워크트리로.

**핵심 습관: 브랜치는 짧게 살린다.** 반나절~하루 안에 끝나 merge되고 사라지는 브랜치가
이상적. 이게 아래 "main 동기화" 고민을 대부분 없애줍니다.

## 4. ★ 창님의 질문: "각 브랜치는 늘 main을 pull 해야 하나?"

**짧은 답: 아니요, 늘 할 필요 없습니다. 대부분은 안 해도 됩니다.**

### 왜 안 해도 되나
브랜치는 "만든 순간의 main"에서 갈라져 나온 **독립된 작업 공간**입니다.
내 작업이 끝나 **PR로 merge할 때, main의 최신 상태와 합쳐지는 건 그 순간 자동으로**
일어납니다. 그래서 짧게 살다 가는 브랜치는 중간에 main을 끌어올 이유가 없어요.
main이 그 사이 좀 바뀌었어도, **내가 건드린 파일과 겹치지 않으면 충돌은 없습니다.**

### 그럼 언제 동기화가 "필요"한가 (딱 3가지)
1. **브랜치가 오래 산다** (며칠씩). 그동안 main이 많이 앞서가면, 내 브랜치가 옛 코드 위에
   지어져 나중에 합치기 어려워짐 → 주기적으로 최신 main을 끌어와 얹어준다.
2. **다른 에이전트가 main에 넣은 결과물이 내 작업에 지금 필요하다**
   (예: 세션 A가 공용 CSS를 고쳐 merge했고, 세션 B가 그걸 써야 함).
3. **PR을 올렸더니 GitHub가 "conflict"라고 막는다** → 합치기 전에 내 쪽에서 풀어야 함.

### 동기화하는 실제 명령 (내 워크트리 폴더 안에서)
`git pull`을 그냥 치는 게 아니라, **"최신 main을 받아 → 내 브랜치에 얹는다"** 2단계입니다:

```bash
git fetch origin              # ① 원격의 최신 정보(=최신 main 포함)를 받아온다. 내 파일은 안 건드림
git rebase origin/main        # ② 내 커밋들을 "최신 main 위로" 옮겨 다시 쌓는다
#   또는
git merge  origin/main        # ② 최신 main을 내 브랜치로 끌어와 합친다 (합침 커밋이 하나 생김)
```

> `git pull`은 사실 `fetch + merge`를 한 번에 하는 것.
> `git pull --rebase`는 `fetch + rebase`. 아래 rebase/merge 차이를 보고 하나를 택하면 됩니다.

### rebase vs merge — 뭐가 다르나 (초심자 감각)
| | `rebase` | `merge` |
|---|---|---|
| 히스토리 모양 | **직선**으로 깔끔 (내 커밋을 최신 위로 옮겨 다시 씀) | 갈래가 보이고 "합침 커밋"이 생김 |
| 장점 | 로그가 읽기 쉬움 | 실제 있었던 일을 그대로 보존 |
| 주의 | **이미 push해서 남과 공유 중인 브랜치는 rebase 금지** (커밋을 다시 써서 꼬임) | 로그에 합침 커밋이 늘어남 |
| 언제 | 혼자 쓰는 짧은 브랜치 | 여럿이 공유하는 브랜치 |

**혼자 하는 짧은 브랜치라면 `fetch` + `rebase`가 깔끔합니다.**

## 5. PR로 합치기 (main 최신 만들기)

```bash
git push -u origin feature/hero-video     # 내 브랜치를 원격에 올림
gh pr create --fill                        # PR 생성 (제목/본문은 영어 — 우리 규칙)
gh pr merge --squash                       # 검토 후 합치기 (squash = 커밋들을 하나로 눌러 합침)
```

> 우리 규칙: **커밋 메시지·PR 설명·`.github/workflows/`는 전부 영어.**
> 설명 문서(이 파일 같은)는 한국어.

merge되면 **main이 최신**이 됩니다. 이제 다른 워크트리들이 그 결과가 필요하면,
그때 4단계의 `fetch` + `rebase`로 각자 끌어오면 됩니다 — **필요할 때만.**

## 6. 워크트리 정리 (merge 끝난 폴더 치우기)

```bash
git worktree remove ../physical-spark-hero-video   # 폴더 제거
git branch -d feature/hero-video                    # 합쳐진 브랜치 삭제
git worktree prune                                  # 유령 워크트리 기록 청소
```

## 7. 자주 밟는 함정

- **같은 파일을 두 에이전트가 동시에 편집** → 각자 폴더에선 문제없지만, 나중에 PR merge 때
  충돌이 남. 되도록 **작업을 파일 단위로 안 겹치게** 나누면 충돌이 거의 안 생김.
- **`node_modules`·빌드 산출물·`.env`는 워크트리끼리 공유 안 됨** (폴더가 다르니까).
  새 워크트리에선 필요한 설치/환경파일을 다시 챙겨야 할 수 있음. (이 repo는 정적 사이트라
  대체로 해당 없음.)
- **같은 브랜치를 두 워크트리에서 동시에 checkout 못 함** — git이 막습니다. 브랜치 하나당 폴더 하나.
- **main 폴더에서 실수로 작업** → main이 지저분해짐. main 폴더는 "통합/배포 전용"으로만.


## 9. 우리 하우스 규칙 (창님이 확정)

여러 방식이 다 "맞지만", 팀이 **하나로 통일**해 두면 매번 고민이 사라집니다.
아래 표를 채워서 우리만의 규칙으로 확정해 주세요.

아래는 소규모 팀·솔로 개발의 **업계 표준(industry standard)** 기본값입니다.
배우는 단계이니 일단 이대로 쓰고, 익숙해지면 바꿔도 됩니다.

| 항목 | 우리 규칙 |
|---|---|
| **main 동기화 방식** | feature 브랜치는 `git fetch origin && git rebase origin/main`로 최신 main을 얹는다. **단, 이미 push해서 공유 중인 브랜치는 rebase 금지** → 그땐 `git merge origin/main`. PR을 main에 합칠 땐 `--squash`(커밋을 하나로 눌러 합침). |
| **브랜치 이름 규칙** | `<종류>/<주제>` 형식, 영어·kebab-case(소문자+하이픈). 종류 접두사: `feature/`(새 기능), `fix/`(버그 수정), `docs/`(문서), `chore/`(잡일·설정). 예) `feature/hero-video`, `fix/game-death-bug`, `docs/worktree-guide` |
| **브랜치 수명 상한** | **하루 안에 끝내 merge**하는 것을 기본으로. 길어도 2~3일; 그보다 커지면 작업을 잘게 쪼갠다. (브랜치가 짧을수록 4번의 동기화 고민이 사라진다.) |

> 왜 이게 표준인가: 짧은 브랜치 + rebase + squash merge = **main 히스토리가 한 줄로 깔끔**하고,
> 각 PR이 "하나의 완결된 변경"으로 남아 나중에 되돌리기·추적이 쉽습니다. GitHub·대부분의
> 스타트업이 쓰는 "trunk-based development(main을 중심 줄기로 삼고 짧은 가지만 치는 방식)"의 기본형이에요.

## 10. 원어민은 이 단어들을 이렇게 느낀다 (git 동사 어휘 감각)

git 명령어는 **개발 용어가 아니라 그냥 평범한 영어 일상 단어**예요. 원어민은 이 단어를
보면 개발과 무관한 일상 장면이 먼저 떠오르고, 명령의 동작이 그 장면과 똑같아서 자연스럽게
이해합니다. 그 "일상 장면"을 알면 창님도 단어만 보고 동작이 그려집니다.

### 핵심 5개 (이것만 몸에 붙여도 충분)

- **fetch** — 강아지에게 "Go fetch!"(가서 물어와!) 할 때 그 fetch. **"가서 가져오다"**.
  원격 저장소에 가서 최신 정보를 물어옵니다. 단, **물어와서 바닥에 내려놓을 뿐 내 작업엔 안 얹어요.**
  그래서 fetch는 안전합니다 — 내 파일은 하나도 안 바뀝니다.

- **pull** — **"당기다"** (문을 pull 해서 당겨 여는 그 pull). 원격 것을 가져와 **내 쪽으로 당겨 합칩니다.**
  즉 `pull = fetch(가져오기) + merge(합치기)`. "가져와서 바로 내 것에 붙이기"라 fetch보다 한 발 더 나감.

- **push** — **"밀다"** (pull의 반대, 문을 밀어 여는 push). 내 커밋을 원격으로 **밀어 올립니다.**
  "내가 만든 걸 저 위(GitHub)로 밀어 보낸다"는 방향 감각.

- **merge** — **"합치다·융합"**. 고속도로에서 두 차선이 하나로 합류할 때 "lanes merge"라고 해요.
  두 갈래(브랜치)를 하나로 합칩니다. 합류 지점에 표시(합침 커밋)가 하나 남습니다.

- **rebase** — **`re`(다시) + `base`(밑바탕)** = **"밑바탕을 다시 깐다"**.
  내 작업은 "예전 main"이라는 바닥 위에 지어졌는데, 그 **바닥을 최신 main으로 갈아 끼우고
  내 벽돌(커밋)을 그 위에 다시 쌓는** 겁니다. 결과적으로 "내가 처음부터 최신 위에서 작업한 것처럼"
  히스토리가 정리돼요. 그래서 줄이 직선으로 깔끔해집니다.

  > 원어민 감각: "rebase"는 군대·회사에서 "본거지(base)를 옮긴다"는 뜻으로도 써요.
  > 작업의 **출발점을 옮긴다**는 이미지가 정확히 그겁니다.

### 자주 나오는 나머지 단어들

| 단어 | 일상 뜻 | git에서 |
|---|---|---|
| **commit** | (약속을) 확정하다, 각오하고 정하다 | 변경을 하나의 저장점으로 **확정**한다 |
| **branch** | 나뭇가지 | main 줄기에서 갈라진 **가지** |
| **checkout** | (도서관 책을) 대출해 꺼내오다 | 그 브랜치/버전을 **꺼내 와 지금 작업 대상으로 삼는다** |
| **stash** | 몰래 따로 챙겨 숨겨두다 | 하던 작업을 잠깐 **서랍에 넣어둔다** (나중에 꺼냄) |
| **clone** | 복제(하다) | 저장소 전체를 **통째로 복제**해 내 컴퓨터로 |
| **origin** | 근원·출처·원산지 | 내가 clone해 온 **원본 원격 저장소의 별명** (보통 GitHub) |
| **upstream** | (강의) 상류·위쪽 | 물이 내려오는 **원본 쪽**. `origin/main`이 보통 upstream |
| **HEAD** | 머리·맨 앞 | **지금 내가 서 있는 위치**(현재 커밋)를 가리키는 화살표 |
| **fast-forward** | (녹화 테이프) 빨리감기 | 갈래 없이 포인터만 **쭉 앞으로** 감아 합치는 것 |
| **conflict** | 충돌·다툼 | 같은 줄을 서로 다르게 고쳐 git이 **누구 말이 맞는지 못 정하는 상태** |
| **diff** | difference(차이)의 준말 | 두 버전의 **바뀐 부분만** 뽑아 보여주기 |
| **stage / add** | 무대에 올리다 | 커밋하기 전 변경을 **무대(대기실)에 올려**둔다 |

### 한 문장 스토리로 꿰기

> 원격에서 최신을 **fetch**(물어와)하고, 내 브랜치의 **base**를 최신으로 **re**-깔아(**rebase**),
> 내 **commit**들을 그 위에 다시 쌓은 뒤, 원격으로 **push**(밀어 올려)해서
> PR로 main에 **merge**(합류)한다.

이 한 줄이 4~5번 섹션의 전부입니다. 단어의 일상 뜻이 곧 명령의 동작이에요.
