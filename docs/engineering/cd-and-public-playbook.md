## 0. 시작된 질문

> "지금 이 서비스가 `ssh bithabit`에서 어떻게 서비스 중인가?
> GitHub에 push하면 그게 CD를 통해 반영됐으면 한다.

마지막 문장이 이 문서 전체의 출발점이다. 그리고 그건 **거의 모든 사람이 하는 오해**다.

## 1. 쿠버네티스는 GitHub을 모른다

이게 핵심이다. 쿠버네티스(k3s)가 하는 일은 딱 하나 —
**선언된 상태를 계속 참으로 유지하는 것**이다.

> "nginx 파드 1개가 떠 있어야 한다"

이 선언을 지키기 위해 파드가 죽으면 다시 띄우고, 노드가 재부팅되면 복구한다.
하지만 **"GitHub에 새 코드가 올라왔는지" 확인하는 기능은 쿠버네티스에 아예 없다.**
그런 코드가 없다. 앞으로도 생기지 않는다.

그래서 **CD는 항상 클러스터 "바깥"의 누군가가 한다.**
"새 코드가 왔다"고 알려주는 주체가 반드시 따로 필요하고, 우리 경우엔 그게
**GitHub Actions**다.

```
git push origin main
   ↓
GitHub Actions  ← GitHub을 지켜보는 유일한 존재
   ↓  SSH (배포 전용 키)
bithabit 서버: 배포 스크립트 실행
   ↓  git reset --hard origin/main
~/workspace/physical-spark/   ← 파일이 바뀜
   ↓  (파드가 이 폴더를 그대로 읽고 있음 — hostPath 마운트)
physical-spark.bit-habit.com  ← 즉시 반영
```

k3s의 역할은 "파드를 살아있게 하고, 443으로 들어온 요청을 거기 꽂아준다"까지.
**코드 갱신은 전적으로 Actions + 스크립트의 몫이다.**

### "업계 표준"이라는 말의 함정

"k3s 업계 표준으로 하고 싶다"고 할 때 흔히 떠올리는 건 **GitOps**(ArgoCD, Flux)다.
그런데 그것도 **쿠버네티스가 해주는 게 아니라, 추가로 설치하는 도구**다.
클러스터 *안*에 에이전트를 심어서 그 에이전트가 Git을 폴링하게 만드는 것.

| 방식 | 코드 갱신 주체 | 우리 |
|---|---|---|
| **Push 방식** (CI → SSH/kubectl) | CI가 클러스터를 **민다** | ✅ 여기 |
| **GitOps** (ArgoCD/Flux) | 클러스터 안 에이전트가 Git을 **당긴다** | ❌ 미도입 |
| **이미지 빌드** | CI가 이미지를 굽고 rollout | `auth/`만 |

Push 방식도 명백한 업계 표준이다. GitOps가 "더 최신"일 뿐 "더 옳은" 건 아니다.
GitOps는 **클러스터가 여러 개거나 팀이 여러 명일 때** 진가가 나온다.
단일 노드·1인 개발에서는 관리할 부품만 늘어난다.

> 참고: `bit-habit-infra` 레포에 ArgoCD 매니페스트가 있지만 **설치되어 있지 않다.**
> (클러스터에 `argocd` 네임스페이스 자체가 없다.) 그 레포 README에 명시해뒀다.

## 2. 발견한 구멍 세 개

배포는 **이미 잘 돌고 있었다.** push → 11초 → 라이브. 최근 실행 전부 success.
문제는 다른 데 있었다.

### 구멍 1 — 배포 스크립트가 버전 관리 밖에 있었다

`physical-spark-deploy.sh`가 **서버에만 존재하는 untracked 파일**이었다.
배포 로직을 고치려면 SSH로 들어가 손으로 편집해야 했고, 그 이력은 어디에도 안 남았다.
배포를 자동화하면서 정작 배포 스크립트만 수동이었던 셈.

### 구멍 2 — 매니페스트 변경이 조용히 무시됐다

스크립트는 `git pull`과 auth 이미지 빌드만 했다. **`kubectl apply`를 하지 않았다.**
그래서 `k8s/site.yaml`을 고쳐서 push하면 — 파일은 서버에 도착하지만
**클러스터는 영원히 모른다.** 아무 에러도 안 난다. 그냥 아무 일도 안 일어난다.
가장 위험한 종류의 버그다.

### 구멍 3 — infra 문서에 "코드가 들어오는 길"이 없었다

`infra.bit-habit.com`은 요청이 *들어오는* 길(브라우저 → Traefik → 앱)을
6단계로 친절히 설명하면서, 코드가 *나가는* 길은 통째로 비어 있었다.
그러니 읽는 사람이 "k3s가 알아서 하나 보다"라고 오해하는 게 당연했다.

## 3. 고친 방법

### 배포 로직 자체를 배포 가능하게

SSH forced command는 `authorized_keys`에 **경로가 고정**돼 있어서 서버 파일을
완전히 없앨 수는 없다. 그래서 그 파일을 **최소한의 부트스트랩**으로 축소했다.

```bash
# 서버: ~/workspace/physical-spark-deploy.sh (4줄)
cd /home/ubuntu/workspace/physical-spark
before=$(git rev-parse HEAD 2>/dev/null || echo none)
git fetch origin -q && git reset --hard origin/main -q
exec ops/deploy.sh "$before"      # ← 바통을 레포 안 스크립트에 넘긴다
```

먼저 pull하고, **레포 안의 `ops/deploy.sh`에 바통을 넘긴다.**
이제 배포 절차를 바꾸려면 `ops/deploy.sh`를 고쳐서 push하면 된다.
**배포 로직이 배포된다.**

> `exec`를 쓴 이유: 부트스트랩 프로세스를 새 스크립트로 **교체**해버린다.
> 자식 프로세스가 아니라서 종료 코드와 stdout이 그대로 SSH 세션을 타고
> GitHub Actions 로그로 흘러간다.

GitOps 도구들도 원리는 같다. **부트스트랩은 손으로 심고, 그 뒤는 전부 Git이 굴린다.**

### 매니페스트를 매번 apply — 자가 치유

```bash
sudo -n k3s kubectl apply -f k8s/site.yaml -f auth/k8s/
```

"바뀐 것만 apply"가 아니라 **무조건 매번**이다. 이유:

- `kubectl apply`는 **멱등(idempotent)** 하다. 바뀐 게 없으면 `unchanged` 한 줄이 전부.
- 매번 적용하면 배포가 **자가 치유(self-healing)** 된다.
  누군가 서버에서 클러스터를 손으로 만졌어도 다음 배포 때 git 상태로 되돌아온다.
- **"파일이 진실이고, 돌아가는 클러스터는 진실이 아니다."**

이건 GitOps의 핵심 사상 그 자체다. 우리는 ArgoCD를 안 깔았지만,
그 사상의 실질적 이득(*선언된 상태로의 수렴*)은 **한 줄로** 가져왔다.

### 비용 표 — 무엇을 고치면 무엇이 필요한가

| 고친 것 | 필요한 일 | 왜 |
|---|---|---|
| `site/`, `docs/`, 아무 `.md` | **없음** — `git pull`로 즉시 라이브 | 파드가 hostPath로 체크아웃을 직접 읽음 |
| `viewer/server.py` | 파드 재시작 (~5초) | 파이썬은 코드를 시작 시점에 메모리로 올림 |
| `auth/` | 이미지 빌드 + rollout (~1분) | 진짜 컨테이너 이미지 |
| `k8s/`, `auth/k8s/` | `kubectl apply` | 어차피 매번 적용됨 |

## 4. 플레이북 공개 — 랜딩과 한 프로세스로

`viewer/`는 원래 **로컬 전용 도구**였다. `127.0.0.1:8766`에 뜨는 개인 문서 편집기.
"building in public"이 이 프로젝트의 정체성인데, 정작 그 과정 기록은 로컬에만 있었다.

### 왜 프로세스를 합쳤나

`viewer/server.py`를 열어보니 **이미 `site/` 정적 파일을 서빙하고 있었다.**
랜딩과 뷰어를 한 프로세스가 감당할 준비가 절반쯤 돼 있던 것. 라우트만 뒤집었다.

```
/          랜딩 페이지 (site/index.html)
/docs      플레이북 — 레포의 모든 .md
/api/*     플레이북의 파일 트리·내용 API
그 외      site/ 정적 파일 (courses/, assets/ …)
```

### 이미지를 굽지 않았다

`server.py`는 **표준 라이브러리만** 쓴다(의존성 0). 그러니 커스텀 이미지를 만들 이유가 없다.
공식 `python:3.12-alpine`을 그대로 쓰고, 레포를 hostPath로 마운트한 뒤 실행한다.

**Dockerfile 없음. 레지스트리 없음. 빌드 단계 없음.**
문서를 고치면 `git pull`만으로 즉시 반영된다 (파이썬이 매 요청마다 디스크를 읽으니까).

## 5. 공개하면서 막은 것 — 로컬 도구를 그대로 올리면 안 되는 이유

`viewer/server.py`에는 **인증 없는 쓰기 API**가 있었다.

```python
def do_POST(self):
    if url.path == "/api/save":
        ...  # 파일을 덮어쓴다. 인증 없음.
```

`127.0.0.1`에 묶인 개인 편집기일 땐 **완벽히 합리적인 설계**였다.
내 컴퓨터에서 나만 쓰니까. 그런데 바인딩 주소가 `0.0.0.0`으로 바뀌는 순간,
**코드는 한 글자도 안 바뀌었는데 취약점이 된다.** 아무나 문서를 덮어쓸 수 있다.

> **교훈:** 로컬 도구를 그대로 배포하는 건 거의 항상 위험하다.
> 위험한 건 코드가 아니라 **코드가 기대던 전제**다.

### 방어를 두 겹으로

| 계층 | 방어 |
|---|---|
| **애플리케이션** | `READ_ONLY=1` → `/api/save`가 **403**을 반환 |
| **커널** | hostPath 마운트가 `readOnly: true` — 쓰기 자체가 불가능 |
| ~~클라이언트~~ | Save 버튼 숨김 — **보안 아님.** UI 정리용일 뿐 |

서로 다른 계층의 방어는 **같은 실수로 동시에 뚫리지 않는다.**
환경변수를 실수로 빼도 파일시스템이 막는다.

클라이언트 가드는 보안이 아니라는 점이 중요하다. 버튼을 숨겨도
`curl -X POST`는 그대로 날아간다. **진짜 경계선은 언제나 서버에 있다.**

실제 공개 서버를 공격해서 확인했다:

```console
$ curl -X POST https://physical-spark.bit-habit.com/api/save \
    -d '{"path":"README.md","content":"HACKED"}'
{"error": "read-only server — editing is local only"}   → HTTP 403
```

### 403이냐 404냐

거절 상태 코드는 진짜 설계 결정이다.

- **403 Forbidden** — "엔드포인트는 있지만 넌 못 쓴다." 정직하고 디버깅하기 쉽다.
- **404 Not Found** — 엔드포인트의 존재 자체를 숨긴다(security through obscurity).

**403을 골랐다.** 숨길 게 없는 공개 시스템(공개 레포·공개 문서)에서 404로 감추는 건
**공격자는 못 속이고 미래의 나만 헷갈리게** 만든다.
403 + 이유 문구(`read-only server — editing is local only`)는 나중에
로컬/배포를 헷갈릴 때 스스로 답해준다.

## 6. 개인 정보는 구조적으로 안전하다

문서를 통째로 공개하면서 가장 걱정한 부분. 그런데 확인해보니 **필터링이 필요 없었다.**

- 서버는 `git reset --hard origin/main`으로만 코드를 받는다.
- 따라서 **gitignore된 파일은 서버에 존재할 수조차 없다.**
- `CLAUDE.md`(창업자 개인 컨텍스트 — 이직 상황, 우선순위 등)는 gitignore돼 있다.

```
로컬 문서: 30개   ← CLAUDE.md 포함
서버 문서: 29개   ← 없음
```

**`.gitignore`가 사실상 "공개 경계선" 역할을 하고 있다.**
배포 파이프라인이 `git reset --hard`라는 사실이 보안 속성을 하나 **공짜로** 만들어준 셈.
차단 목록을 관리하는 게 아니라, **구조가 보장한다.**

## 7. 배포하다 한 번 깨진 것 — readiness probe가 사이트를 지켰다

첫 배포에서 파드가 뜨긴 했는데 **Ready가 안 됐다.**

원인: `HOST` 환경변수를 상수로 선언해놓고 **정작 서버를 띄우는 줄에서 안 썼다.**

```python
HOST = os.environ.get("HOST", "127.0.0.1")   # 선언은 했는데
...
server = ThreadingHTTPServer(("127.0.0.1", PORT), Handler)   # ← 하드코딩 그대로
```

컨테이너 안에서 `127.0.0.1`에 묶이니 바깥에서 닿을 수 없었고, probe가 실패했다.

**그런데 사이트는 1초도 안 죽었다.**

- probe가 실패하자 k8s는 새 파드를 **서비스에 넣지 않았다.**
- **낡은 nginx 파드가 계속 트래픽을 받았다.**
- 배포가 실패하는 동안 사용자는 아무것도 못 느꼈다.

그게 롤링 업데이트의 계약이다. probe가 없었다면 k8s는 "컨테이너가 떴으니 Ready"로 판단하고
**죽은 파드로 트래픽을 흘려보냈을 것이다** — 랜딩 페이지가 통째로 다운됐을 것이다.

곁들인 교훈: 로그가 **비어 있었다.** 파이썬은 stdout이 파이프일 때 블록 버퍼링을 해서
`kubectl logs`에 아무것도 안 보였고, 그게 진단을 늦췄다. `print(..., flush=True)`로 고쳤다.

## 8. 곁다리로 정리한 것들 (bit-habit-infra)

- **`infra.bit-habit.com`에 "How code gets here" 챕터 추가** — 요청이 들어오는 길만
  설명하고 코드가 나가는 길은 비어 있던 구멍. "⚠️ Kubernetes does not watch GitHub"
  콜아웃으로 시작한다.
- **physical-spark 등록** — 상태 모니터링(Gatus), 서비스 표, README.
  최신 서비스인데 infra 레포에 **단 한 글자도 없었다.**
- **두 계층 구분 명시** — 앱 콘텐츠는 자동(Actions), 인프라 매니페스트는 수동(`kubectl apply`).
  잘못된 Ingress 하나가 20개 사이트를 한 번에 죽일 수 있으니 사람이 개입하는 게 맞다.
- **ArgoCD 모순 해소** — 매니페스트는 있는데 설치는 안 됨. 지우는 대신
  "참고용, 설치 안 함 + 왜 안 쓰는지"를 명시했다.

## 요약 — 오늘 얻은 원칙

1. **쿠버네티스는 GitHub을 모른다.** CD는 항상 클러스터 바깥에서 온다.
2. **파일이 진실이고, 돌아가는 클러스터는 진실이 아니다.** 매번 `apply` → 자가 치유.
3. **배포 로직도 배포돼야 한다.** 서버에만 있는 스크립트는 이력이 없다.
4. **로컬 도구를 그대로 배포하지 마라.** 위험한 건 코드가 아니라 코드가 기대던 전제다.
5. **클라이언트 가드는 보안이 아니다.** 진짜 경계선은 언제나 서버에 있다.
6. **구조가 보장하는 안전이 가장 좋은 안전이다.** gitignore가 곧 공개 경계선.
7. **가장 위험한 버그는 에러를 안 낸다.** apply 안 하던 매니페스트처럼.
