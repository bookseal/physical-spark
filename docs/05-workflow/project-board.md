# GitHub Project 보드 — 지금 어떻게 굴러가나

> **대상:** 이 보드를 처음 보는 사람 (그리고 6개월 뒤의 나)
> **보드:** https://github.com/users/bookseal/projects/4
> **관련:** [개발 흐름 전체](github-workflow.md) — 이슈·브랜치·PR·머지의 한 바퀴

---

## 0. 한 문장

**보드는 "지금 뭘 하고 있나"를 보여주는 화면이고, 카드는 손으로 옮기지 않는다. 이슈와 PR의 상태가 카드를 자동으로 옮긴다.**

---

## 1. 이 보드가 왜 있나

`.md` 문서는 **"무엇을 왜"**를 담는다. 하지만 문서만으로는 이 질문에 답할 수 없다:

> **"그래서 지금 무슨 일을 하고 있는데?"**

이슈가 그 답이고, 보드는 그 이슈들을 **한 화면**에 놓은 것이다.

그리고 이 저장소에서는 보드가 하나 더 하는 일이 있다 — **트랙션.**
커밋 20개는 "코드를 썼다"이고, 이슈 20개 + PR 20개는 **"문제를 정의하고, 결정하고, 검증하며 만들었다"**이다.
[building in public에서 building은 코드가 아니라 과정이다.](github-workflow.md#-트랙션--이게-우리에겐-특히-크다)

---

## 2. 5개의 칸 — 카드는 여기를 지나간다

| 칸 | 뜻 | **언제 자동으로 여기 오나** |
|---|---|---|
| **Backlog** | 적어는 뒀다. 아직 안 할 것 | (수동) 지금은 안 할 걸 밀어둘 때 |
| **Todo** | 할 것. 아직 시작 안 함 | **이슈를 새로 만들면** |
| **In Progress** | 지금 하고 있음 | (수동) 내가 붙잡을 때 |
| **In Review** | **PR이 열려 있음** | **그 이슈를 닫는 PR이 열리면** |
| **Done** | 끝남 | **이슈가 닫히거나, PR이 머지되면** |

**핵심: `Todo` → `In Review` → `Done` 은 손대지 않아도 저절로 간다.**
사람이 손으로 옮기는 건 `In Progress` 하나뿐이다 (그리고 그것조차 안 해도 된다).

> **왜 이게 중요한가:** 손으로 옮기는 보드는 반드시 죽는다.
> 며칠만 지나면 보드와 실제가 어긋나고, 어긋난 보드는 아무도 안 본다.
> **자동화가 곧 보드의 생명이다.**

## 3. 실제 사용법 — 하루 흐름

### 새 일이 생겼다
```bash
gh issue create --title "..." --body "..."
```
→ **자동으로 보드에 `Todo`로 올라온다.** (Auto-add 워크플로)
보드에 손댈 일 없음.

### 그 일을 시작한다
```bash
git switch -c feat/눈-달기
# ... 작업 ...
gh pr create --title "..." --body "Closes #1"      # ← 이 한 줄이 전부다
```

> ## ⚠️ `Closes #1` — 이게 보드를 굴리는 유일한 장치다
>
> PR 본문에 **`Closes #이슈번호`**를 쓰면:
> 1. PR과 이슈가 **영구히 연결**된다
> 2. 이슈 카드가 **자동으로 `In Review`**로 간다
> 3. PR을 머지하면 **이슈가 자동으로 닫히고 `Done`**으로 간다
>
> **이 한 줄을 빼먹으면 자동화가 하나도 작동하지 않는다.**
> (`Fixes #1`, `Resolves #1` 도 같은 뜻)

### 머지한다
```bash
gh pr merge 13 --squash
```
→ 이슈 자동으로 닫힘 → 카드 **자동으로 `Done`**.

**보드를 열어서 마우스로 뭘 끌어당길 일이 없다.** 그게 정상이다.

## 5. ⚠️ 지금 자동화가 꺼져 있다 — 켜야 한다

**솔직한 기록.** 보드를 만들 때 `In Review`와 `Backlog` 칸을 나중에 추가했는데, GitHub은 칸을 **이름이 아니라 ID로** 기억한다. 칸을 다시 정의하면서 **ID가 새로 발급됐고, 그 ID를 가리키던 내장 자동화 6개가 전부 무효화되어 꺼졌다.**

**그리고 GitHub API에는 자동화를 켜는 명령이 없다** (`deleteProjectV2Workflow`로 지울 수만 있다). **UI에서 켜야 한다.**

### 켜는 법 — 1분

**https://github.com/users/bookseal/projects/4/workflows**

| 워크플로 | 켜고, 이렇게 설정 |
|---|---|
| **Auto-add to project** | 저장소 `bookseal/physical-spark`, 필터 `is:issue,pr is:open` ← **이게 제일 중요.** 새 이슈·PR을 보드에 자동으로 올림 |
| **Item added to project** | → 상태 `Todo` |
| **Item closed** | → 상태 `Done` |
| **Pull request merged** | → 상태 `Done` |
| **Pull request linked to issue** | → 상태 `In Review` |

이걸 켜기 전까지는 **새 이슈가 보드에 안 올라오고**, 카드도 저절로 안 움직인다.

### 교훈 (다시 겪지 않으려면)

- **칸 구성은 처음에 확정하고 그 뒤로 건드리지 마라.** 이름만 바꿔도 자동화가 조용히 끊긴다.
- 보드 설정은 **git에 없다.** 코드는 전부 버전 관리되는데 보드만 GitHub UI 안에 있어서, 깨져도 되돌릴 소스가 없다.
- 근본 해법: `.github/workflows/`에 [`actions/add-to-project`](https://github.com/actions/add-to-project)를 넣어 **자동 등록을 코드로** 관리하는 것. (단 project 권한이 있는 PAT를 시크릿으로 넣어야 한다 — 기본 `GITHUB_TOKEN`은 사용자 프로젝트에 쓸 수 없다.) → 아직 안 함.

## 6. 자주 헷갈리는 것

**Q. 보드에 카드를 직접 만들 수 있던데?**
만들지 마라. 보드에만 있는 카드(draft)는 **이슈가 아니라서** 브랜치·PR·커밋 어디에도 안 붙는다. 검색도 안 되고, 외부인에게 안 보인다. **항상 이슈로 만들어라.**

**Q. `Backlog`는 언제 쓰나?**
"언젠가 할 것"을 `Todo`에서 빼두고 싶을 때만. `Todo`가 20개를 넘어가면 그 보드는 이미 안 보게 된다.

**Q. 이슈를 손으로 닫아도 되나?**
된다. 닫으면 자동으로 `Done`으로 간다. 다만 **PR로 닫는 게 낫다** — 그래야 "이 이슈가 어떤 코드로 해결됐는지"가 영구히 남는다.

**Q. Plane(`plane.bit-habit.com`)도 쓰는데?**
[역할이 다르다.](github-workflow.md) GitHub Projects = **공개 트랙션**(외부인이 보는 과정). Plane = **비공개 개인 할 일**(로봇팔 주문 추적 등).
